- [RESOLVIDO — ADR-013] Porque precisamos ter eng/script.pl e um wrapper eng/script.ps1
  que chama eng/script.pl? Usar como padrão `perl eng/script.pl` não parece uma boa
  ideia já que temos o Perl garantido sempre?
  → Wrapper .ps1 removido. `carton exec perl eng/script.pl` é o único comando, igual
    em qualquer plataforma. O wrapper antigo nem sequer usava `carton exec` (bug real).

- [RESOLVIDO — ADR-016, revisado 2026-07-02] Mojo::Pg::Migrations 5.0 já tem um padrão
  que usa um diretório de migração ao invés de um único arquivo (from_dir). Vale
  revisar se estamos usando as últimas versões das bibliotecas Mojolicious.
  → Adotado `from_dir`. Decisão original (manter o loader customizado via
    `from_string`) foi revertida: usar o mecanismo que a própria biblioteca oferece é
    mais seguro do que manter um utilitário próprio equivalente. `migrations/N/up.sql`
    e `migrations/N/down.sql` substituíram os arquivos únicos `NNN_descricao.sql`; a
    descrição que estava no nome do arquivo agora é um comentário na primeira linha
    de cada `up.sql`/`down.sql`.

- [RESOLVIDO — ADR-019, nova] Vale usar `use Modern::Perl '2025';` por padrão nos
  scripts? Ou `use 5.042` com `use warnings`/`use strict`/`use autodie`/`use utf8`?
  Tivemos problemas de codificação de caracteres.
  → Padrão definido: `use v5.42; use utf8;` (mais `use open ':std', ':encoding(UTF-8)';`
    em scripts que imprimem para o terminal). `use v5.42` já habilita strict/warnings.
    Sem `Modern::Perl` (dependência externa desnecessária). `autodie` fica opcional,
    não é padrão. Aplicado em todo `eng/*.pl`, `lib/**/Model/*.pm`, `lib/**/Job/*.pm`,
    `lib/Stega/Worker/*.pm`, `script/stega` e `t/**/*.t`. Controllers e `lib/Stega.pm`
    não precisaram de mudança — `Mojo::Base ... -strict` já aplica utf8 automaticamente.

- [RESOLVIDO — ADR-014] Qual repositório oficial do berrybrew — dnmfarrell ou stevieb9?
  → `stevieb9/berrybrew` é o oficial e ativamente mantido (o próprio README do
    repositório original aponta para lá). Instalação: `berrybrewInstaller.exe` ou
    clone + `bin\berrybrew.exe config` (como Administrador). Atualizado em
    DEVELOPMENT.md e README.md.

- [RESOLVIDO — ADR-005] cpanfile tem questões de versões parciais (>=, ~=, etc.)?
  `requires 'Mojolicious', '9.0';` baixa exatamente 9.0 ou a mais recente?
  → Número isolado = versão mínima (`>= 9.0`); o `carton install` resolve para a mais
    recente compatível, e é o `cpanfile.snapshot` (não o `cpanfile`) que fixa a versão
    exata. Sintaxe de intervalo (`>= X, < Y`) usada quando necessário — já é o padrão
    usado aqui para travar `Mojolicious::Plugin::OpenAPI`/`JSON::Validator` abaixo da
    versão com a dependência XS incompatível com Perl 5.42.

- [RESOLVIDO — ADR-009] Usar Mojolicious::Plugin::OpenAPI só, ou também
  Mojolicious::Plugin::OIDC/OAuth2 para o Keycloak?
  → Mantido o fluxo manual (redirect + troca de código + validação via JWKS com
    Crypt::JWT) já implementado em Stega::Controller::Auth. Os plugins foram avaliados
    e rejeitados: a API já precisa de validação local via JWKS de qualquer forma (só
    recebe `Authorization: Bearer`), e o fluxo web tem ~40 linhas auditáveis que já
    resolvem o caso conhecido do `id_token` sem `realm_access.roles`.

- [RESOLVIDO — ADR-011 + ADR-020] Testes majoritariamente de integração (acessam
  banco). Queria testes unitários puros para regras de negócio, com mocks. Item
  aprofundado em 2026-07-02: queria isso para toda funcionalidade, não só permissão —
  ex. "posso criar dois produtos com o mesmo nome/slug?" validado por uma classe de
  domínio que usa um mecanismo abstrato de persistência, mockável em teste.
  → Duas camadas complementares, não uma só:
    1. **Policy** (ADR-011): autorização — `Stega::Domain::TicketPolicy`, já cobre as
       regras de `BUSINESS.md` (quem pode fazer o quê), usada por rotas web e API.
    2. **Domain + Repository** (ADR-020, novo): validação de negócio + execução — uma
       classe por entidade (`Stega::Domain::Product`) recebe um Repository injetado
       (`Stega::Repository::Product`, contrato via `Moo::Role`; implementação real
       `Stega::Repository::Pg::Product`). Em teste, o Repository é trocado por uma
       classe Fake (`Stega::Test::Repository::Product`, em `t/lib/` — não em `lib/`,
       seguindo a convenção Perl de código só-de-teste) em vez de `Test::MockObject`:
       a regra de duplicidade exige estado real entre chamadas (inserir → buscar),
       que `Test::MockObject` exigiria simular manualmente com o mesmo resultado.
       `Test::MockObject` continua reservado para interações pontuais com serviços
       externos (RabbitMQ, HTTP), como já previsto desde ADR-011.
    Piloto implementado em `Product` (`lib/Stega/Domain/Product.pm`,
    `lib/Stega/Repository/{Product.pm,Pg/Product.pm}`,
    `t/lib/Stega/Test/Repository/Product.pm`, `t/unit/domain/product.t`) — resolveu de
    quebra um bug real: `api_create` de produto não tratava violação da constraint
    `UNIQUE` de slug (estourava 500 em vez de 422) e não validava nome duplicado.

    escopo em relação ao piloto: para essas duas entidades o Repository passou a

    havia duas implementações DIVERGENTES do mesmo upsert de usuário a partir do
    JWT — `require_jwt` (toda requisição de API) fazia um `INSERT ... ON
    CONFLICT DO UPDATE` atômico mas descartava se era usuário novo; `_sync_user`
    (só no login web) fazia `SELECT` + `UPDATE`/`INSERT` separados, não-atômico,

    seed (rota depende de sessão web, sem cobertura automatizada — gap
    pré-existente, não introduzido agora). Sem `Domain::User`: sincronizar um
    JWT já validado não tem regra de estado a proteger. Sem Fake de
    `Repository::User`: nenhum teste a consome ainda. Restam com SQL direto só
    `Health` (checagem de conectividade, não é entidade) e `Webhook` (não toca

- [RESOLVIDO 2026-07-04] Autenticação dos endpoints de webhook via
  credenciais administráveis. Estado que motivou o item: `POST /api/v1/
  webhooks/generic` não tinha autenticação nenhuma; `POST /api/v1/webhooks/
  github` validava HMAC-SHA256 contra um único `GITHUB_WEBHOOK_SECRET` em
  variável de ambiente, mas ficava desativado quando o segredo não estava
  configurado (o caso padrão em desenvolvimento) — nenhum dos dois tinha
  proteção real com a configuração padrão. Achado adicional ao investigar:
  a verificação do GitHub **nunca funcionou de verdade** — o código fazia
  `require Digest::HMAC_SHA256`, mas esse módulo não existe no CPAN (o
  `cpanfile` só traz `Digest::HMAC_MD5`/`Digest::HMAC_SHA1`, não
  `_SHA256`); como o `return 1 unless $secret` sempre disparava primeiro
  (segredo nunca configurado), esse `require` quebrado nunca foi exercitado
  — se alguém chegasse a configurar `GITHUB_WEBHOOK_SECRET`, a verificação
  morreria com "Can't locate Digest/HMAC_SHA256.pm", não validaria nada.

  Requisitos levantados pelo usuário para a solução (não só "adicionar uma
  chave"): o admin cadastra credenciais de webhook (segredo/token); pode
  revogar e recriar (rotacionar) o segredo; pode desativar uma credencial;
  só pode **excluir** se não houver nenhum registro vinculado a ela (a
  interface deve indicar quando há vínculo e, nesse caso, mostrar um alerta
  no lugar do botão excluir); toda ação sobre uma credencial (criar,
  rotacionar, ativar/desativar, excluir) precisa de histórico de quem
  executou e quando, visível na interface; toda ação realizada via webhook
  (hoje: criação de ticket e, no GitHub, resolução ao fechar a issue) deve
  registrar qual credencial a autenticou; o identificador da credencial é
  único e imutável após criado.

  Desenho decidido:
  - Tabela `webhook_credentials` (`id UUID` gerado por `gen_random_uuid()` —
    o identificador imutável exposto ao admin/chamador —, `name`, `source`
    ∈ {`github`, `generic`}, `secret`, `is_active`, `created_by`,
    timestamps). `source` existe porque os dois endpoints autenticam de
    formas diferentes (ver abaixo), não porque a entidade em si é
    diferente.
  - Tabela `webhook_credential_audit` para o histórico de ações sobre a
    própria credencial (criada/rotacionada/ativada/desativada/excluída).
    **Sem FK real** para `webhook_credentials.id` (só um UUID solto, com
    `webhook_credential_name` desnormalizado no momento da ação) —
    propositalmente: se a auditoria referenciasse a credencial com FK, ou o
    `ON DELETE` teria que ser `CASCADE` (destruindo o próprio histórico que
    a exclusão deveria registrar) ou a exclusão ficaria bloqueada pela
    própria auditoria (nunca seria possível excluir nada). Histórico de
    ações sobrevive à exclusão da credencial, por design.
  - "Registro vinculado" (o que bloqueia exclusão) = qualquer ticket cujo
    evento `ticket.created` (ou `status.changed`, no caso do fechamento de
    issue no GitHub) tenha `webhook_credential_id` no `payload` JSONB —
    reaproveita a tabela `events` já existente (ADR-016/020), sem migração
    nova em `tickets`. Contagem via `COUNT(DISTINCT ticket_id) FROM events
    WHERE payload->>'webhook_credential_id' = ?`.
  - Segredo mostrado **uma única vez** (na criação e a cada rotação) — a
    sessão web guarda o valor em texto puro temporariamente, a página de
    detalhe da credencial o exibe e o remove da sessão ao ser carregada; um
    reload subsequente não mostra mais o valor.
  - Verificação continua HMAC-SHA256 nos dois endpoints (não um modelo de
    bearer token simples) — decisão forçada pelo próprio GitHub: o produto
    do GitHub só envia `X-Hub-Signature-256` (HMAC), sem um jeito de dizer
    *qual* segredo usou, então o servidor testa a assinatura contra cada
    credencial `source=github` ativa. Já que o servidor precisa recalcular
    o HMAC, o segredo tem que ficar recuperável em texto puro no banco —
    não dá para só guardar um hash (diferente de senha, que só precisa ser
    conferida, nunca recalculada). Isso é uma limitação aceita para esta
    demonstração; um ambiente de produção real ia querer criptografar essa
    coluna em nível de aplicação (`pgcrypto` ou similar), não guardá-la em
    texto puro — registrado aqui como conhecido, não implementado agora.
    Para manter os dois tipos de credencial consistentes (um só modelo),
    `generic` usa o mesmo esquema: o chamador assina o corpo com o segredo
    e informa qual credencial usou via cabeçalho `X-Webhook-Key-Id`, já que
    aqui (diferente do GitHub) o protocolo é nosso e podemos exigir isso.
  - Comparação de assinatura em tempo constante (não `eq` simples) — já que
    a mudança toca justamente o código de verificação, vale corrigir esse
    detalhe de segurança junto.

  Fora do escopo desta rodada, registrado à parte: revisar o que o GitHub
  oferece além do HMAC (rotação com múltiplos segredos aceitos
  simultaneamente do lado do GitHub, IP allowlist dos webhooks) fica para
  quando/se isso for cogitado para produção de verdade — o que está sendo
  implementado agora já cobre o essencial pedido (credencial administrável,
  revogação/rotação, histórico, atribuição). Ainda não é uma ADR do
  repositório central — decisão de demonstração da Stega.

  → Implementado: migrations 10/11 (`webhook_credentials`,
  `webhook_credential_audit`); `Stega::Repository(::Pg)::WebhookCredential` +
  `Stega::Domain::WebhookCredential` (ADR-020); `Stega::Controller::
  WebhookCredential` (admin: `index`/`new`/`create`/`show`/`rotate`/
  `set_active`/`delete`) com templates próprios e link no menu Admin;
  `Stega::Controller::Webhook` reescrito (HMAC contra credencial em banco,
  comparação em tempo constante, `Digest::SHA::hmac_sha256_hex` no lugar do
  `Digest::HMAC_SHA256` que nunca existiu); `Stega::Domain::Ticket::create`/
  `change_status` ganharam `event_extra` opcional para atribuir a credencial
  ao evento; `Stega::Job::ProcessWebhookPayload` migrado para usar
  Domain+Repository (não mais SQL direto) e corrigido para registrar evento
  `status.changed` no fechamento de issue do GitHub (antes não registrava
  nada); `eng/seed.pl` ganhou 2 credenciais de demonstração com segredo fixo
  (só para os roteiros de `TESTING.md`); `t/030_webhooks.t` reescrito para
  autenticar de verdade; `t/unit/domain/webhook_credential.t` novo.

  Achados reais só na validação de ponta a ponta contra a aplicação real em
  Docker (não só lendo o código):
  1. `t/030_webhooks.t` inicialmente falhava — os jobs Minion enfileirados
     nunca eram processados porque não há `minion-worker` rodando durante a
     suíte (só no `docker compose up` real); corrigido chamando
     `$t->app->minion->perform_jobs` logo após cada `POST` que enfileira,
     processando o job de forma síncrona só para o teste.
  2. Depois disso, as asserções sobre o payload do evento (`webhook_credential_id`)
     ainda falhavam com `got: undef` mesmo com o dado correto no banco —
     causa real: o helper `set_auth` deste projeto usa `$t->ua->once(start
     => ...)`, que só anexa o header `Authorization` na *próxima* requisição;
     meus testes chamavam `set_auth` uma vez e faziam duas requisições
     autenticadas em seguida (a segunda saía sem token, sendo rejeitada
     silenciosamente do ponto de vista do `get_ok`, que não checa status por
     padrão). Todo outro teste do projeto já chama `set_auth` antes de cada
     requisição — os meus não seguiam essa convenção.
  3. Testando a UI admin de verdade: tentar excluir uma credencial com
     tickets vinculados corretamente bloqueava a exclusão (o núcleo do
     requisito), mas a caixa de erro no topo da página não aparecia — causa
     real, não relacionada a `$@`: os templates `.ep` deste projeto exigem
     que **toda variável usada como bareword** (`$revealed_secret` no
     template) esteja entre as chaves passadas a `render()` naquela chamada
     específica — a ação `delete` renderizava `show.html.ep` sem passar
     `revealed_secret`, o que quebra a compilação do template com "Global
     symbol requires explicit package name" (500, não uma página em branco).
     Corrigido passando `revealed_secret => undef` explicitamente nesse
     `render()`; a mensagem de erro em si passou a usar `stash('error')`
     declarado uma vez no topo do template (`% my $error = stash('error');`),
     no mesmo padrão já usado para `current_user` nos outros templates.
  Suíte completa revalidada do zero (13 arquivos, 87 testes, `Result: PASS`)
  e todo o ciclo de vida de uma credencial (criar → segredo some ao
  recarregar → rotacionar → desativar/ativar → excluir sem vínculo →
  bloquear exclusão com vínculo, com a mensagem de erro visível) validado via
  `curl` simulando o fluxo OAuth completo (login real no Keycloak) contra a
  aplicação rodando em Docker, e os dois webhooks (`generic`/`github`,
  incluindo o fechamento de issue) validados com assinatura HMAC real
  gerada a partir do segredo da credencial.

- [RESOLVIDO 2026-07-04] Bug real de dupla codificação UTF-8 na escrita de
  colunas JSONB — achado pelo usuário ao navegar na aplicação: o badge "via
  Webhook" no histórico do ticket 4 mostrava "GenÃ©rico Teste 030" em vez de
  "Genérico Teste 030", mas a própria credencial (mesmo nome, lido de uma
  coluna TEXT comum) aparecia correta na interface admin. Confirmado via

  → Causa raiz: todo `Stega::Repository::Pg::*` usava `encode_json($dados)`

  como bind parameter para um placeholder com cast `::jsonb`. Com

  cada caractere acentuado (2 bytes) vira 4 bytes. Confirmado isolando as
  duas formas de escrita lado a lado no mesmo banco: `encode_json(...)` +
  `::jsonb` → corrompido; marcador nativo `{ json => ... }` do Mojo::Pg (sem
  `encode_json` manual, sem cast) → correto. Ironia: a revisão da ADR-017
  feita mais cedo hoje (ver item de JSONB `expand()` acima) documentou esse
  `encode_json + ::jsonb` como "o padrão real da Stega", em contraste com o
  exemplo original da própria ADR (que já mostrava `{ json => ... }`) —
  documentei um bug como se fosse decisão deliberada. ADR-017 revisada de
  novo para refletir a correção.
  → Corrigido em todos os pontos afetados: `Stega::Repository::Pg::Product`
  (`insert`, `update_fields`), `Stega::Repository::Pg::Comment` (`insert`),
  `Stega::Repository::Pg::Ticket` (`insert_ticket`, `record_event`),
  `Stega::Job::CheckSlaBreaches` (mesmo padrão em SQL direto — sem texto
  acentuado nos dados reais desse job, mas mesma classe de bug). Import de
  `encode_json` removido dos arquivos onde ficou sem uso. Não mexido:
  `eng/seed.pl` usa `'{...}'::jsonb` como **literal embutido na própria
  string SQL**, não como bind parameter — confirmado são do bug (a
  `description` do produto do seed, com "ç"/"ã", já batia `octet_length`
  correto antes desta correção); `Stega::Job::SendWelcomeNotification` usa
  `JSON::PP::encode_json` para publicar no RabbitMQ, não para bind de banco
  — contexto diferente, sem o mesmo risco.
  Regressão coberta por dois testes novos (`t/020_products_api.t`,
  `t/010_tickets_api.t`) que criam registros com acentuação em campo JSONB

  **diferente** do bug de dupla codificação UTF-8 resolvido acima: texto com
  caractere acentuado (ex.: "á", "ó", "ç") lido do Postgres via `Mojo::Pg` neste

  tem `Encode::is_utf8() == 1` (flag ligada, "decodificado") mas os bytes
  internos (`unpack('H*', $str)`) mostram **um único byte solto** (`e1` para

  dentro. Isso não é o mesmo bug já corrigido acima (aquele era escrita de

  registrada acima — "`Stega::Job::SendWelcomeNotification` usa
  `JSON::PP::encode_json` para publicar no RabbitMQ ... contexto diferente, sem
  o mesmo risco" — estava **errada**: o risco existe, só que a causa raiz não é
  o `encode_json` em si, é a leitura anterior a ele).

  → Confirmado **não específico** deste caso: reproduz com um script Perl
  isolado (sem Minion, sem RabbitMQ, sem HTTP), usando a mesma
  `Mojo::Pg->new(...)` com `pg_enable_utf8 = -1` do resto da app; reproduz
  tanto com `SELECT ... WHERE id = $1` (estilo posicional) quanto com
  `WHERE id = ?` (estilo padrão) — não é sobre estilo de placeholder; reproduz
  até lendo uma linha de um fixture de teste **pré-existente e não relacionado**

  use v5.42; use utf8; use open ':std', ':encoding(UTF-8)';
  use Mojo::Pg;
  use Encode qw(is_utf8);
  my $pg = Mojo::Pg->new($ENV{POSTGRESQL_URL});
  $pg->options->{pg_enable_utf8} = -1;
  my $row = $pg->db->query("SELECT 'á'::text AS x")->hash;
